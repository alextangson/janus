# P2.3 — confirm 闭环 设计文档

- **日期**:2026-06-04
- **状态**:已通过设计评审,待实现
- **所属**:Phase 2 第 3 个子项目;P2.1(读设备)、P2.2(执行)已合并 `main`

## 1. 目标

把 `confirm` 从"原样返回"升级为完整的**待确认 → 确认后执行**模型(对 stub;真确认通道留到接真 HA)。

## 2. 设计决策(评审锁定)

- **无状态**:`Controller` 产出"需要确认 + 一句话术 + 待执行 Decision",由调用方(将来的 HA 集成层)在对话轮次间持有该 pending,再调 `confirm(decision, approved)` 收尾。挂起状态属于集成层,不进核心。
- **DRY**:抽出私有 `_execute(decision)`,`allow` 与"确认通过"共用同一段执行 + fail-safe。
- `Outcome` 加 `needs_confirmation: bool = False`、`prompt: str | None = None`。

## 3. 行为

| 入口 | verdict / 入参 | 结果 |
|---|---|---|
| `handle()` | `allow` | 直接执行 → `Outcome(executed=True)` |
| `handle()` | `confirm` | 不执行 → `Outcome(executed=False, needs_confirmation=True, prompt=...)` |
| `handle()` | `reject` | 不执行 → `Outcome(executed=False)` |
| `confirm(decision, True)` | 用户同意 | 执行 → `Outcome(executed=True)`(或失败 → fail-safe `error`) |
| `confirm(decision, False)` | 用户否决 | 不执行 → `Outcome(executed=False)` |

话术(v1 简版):`f"确认执行「{operation} → {device_id}」?{reason}"`。

## 4. 测试(stub,零网络)

- `handle()` 遇 `confirm` → `needs_confirmation=True`、有 `prompt`、`executed=False`、**未调 HA**。
- `confirm(decision, True)` → 执行(`call_service` 被调、`executed=True`)。
- `confirm(decision, False)` → 未执行。
- 确认后执行抛错 → fail-safe 捕获(`executed=False` + `error`)。
- e2e:真 engine 把"开锁"判成 `confirm` → `confirm(True)` → 执行。
- 现有 `allow`/`reject`/error 测试不回归(`needs_confirmation` 默认 False)。

## 5. 成功标准

confirm 流程测试全绿 + 全套零回归;`Controller` 仍无持久状态;执行仍只经 `ha_client`。

## 6. 非目标(留后)

真确认通道(HA 对话/通知/语音)· 服务端 pending 存储 + 超时 · 多轮对话 · 打包(P2.4)。
