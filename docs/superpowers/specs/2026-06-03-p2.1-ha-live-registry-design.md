## 1. Phase 2 全景与本子项目定位

Phase 2 把 Phase 1 验证过的"可靠大脑"接到真 HA。拆成 4 个独立子项目(各自 spec → plan → build):

| 子项目 | 做什么 | 依赖 |
|---|---|---|
| **P2.1 接 HA + 实时设备清单(本文)** | 用从 HA 拉取的真实实体/服务替掉手写 `devices.json` | — |
| P2.2 执行 | 把 `allow` 的决策经 HA service-call 真正执行 | P2.1 |
| P2.3 confirm 闭环 | `confirm` 时真问用户、点头才执行 | P2.2 |
| P2.4 打包成 HA 集成 | custom_component / HACS / config flow | P2.1–2.3 |

**架构基调(评审锁定):先做独立服务 + 调 HA API**(REST/WebSocket),延续 Phase 1「先便宜验证」的打法;之后再包成原生组件(P2.4)。

## 2. 核心设计决策(评审锁定)

| # | 决策 | 选择 | 理由 |
|---|---|---|---|
| D1 | 结合方式 | **独立服务 + 调 HA API** | 最快验证 read+execute+confirm,我们的干净逻辑仍可独立单测;原生组件留到 P2.4。 |
| D2 | 读取方式 | **REST 快照(v1)** | `/api/states` + `/api/services` 足够拉出设备、服务、参数范围、危险所需元数据。WebSocket/实时/area 留后。 |
| D3 | 危险粒度 | **操作级(沿用 Phase 1)** | 只拦标了危险的那个操作。代价:保留 Phase 1b 的"误解析漏放行"洞(靠 temperature=0 + 解析正确率压低;真出现再升级到设备级)。 |
| D4 | 危险来源 | **按域/device_class 自动默认 + 用户覆盖** | 零手工起步、可调。 |
| D5 | 测试 | **录制的样本数据(fixtures)** | 现在没有可连的 HA;真 HA 验证等环境就绪。 |

## 3. 架构与模块

**核心思路:P2.1 插在现有 `Registry` 下方——从 HA 数据"生产"出 `Registry`,不再读手写 JSON。上层(`validator` / `engine` / `parser` / `harness`)一行不改**,因为它们只认 `Device` 模型(Phase 1 对齐 HA 命名埋下的红利)。

新增两个模块:

| 模块 | 职责 | 接口(输入 → 输出) | 碰网络? |
|---|---|---|---|
| `ha_client` | 连 HA REST(URL + token),拉原始数据 | `fetch() → (states: list, services: dict)` | 是(测试注入样本) |
| `ha_mapping` | HA states+services → 我们的 `Device`,套危险默认 + 覆盖 | `map_ha(states, services, overrides) → dict[str, Device]` | 否(纯函数) |

`Registry` 增一个构造入口,与现有 `from_file` 并列:

```python
@classmethod
def from_ha(cls, states, services, overrides) -> "Registry":
    return cls(ha_mapping.map_ha(states, services, overrides))
```

`ha_client` 是唯一碰 HA 网络的地方(对称于 `parser` 是唯一碰模型的地方);`ha_mapping` 是纯函数,喂样本 dict 即可完整单测。

### 数据流

```
HA REST  (/api/states + /api/services)
   → ha_client.fetch()                       [唯一网络边界]
   → ha_mapping.map_ha(states, services, overrides)   [纯函数:过滤 + 映射 + 危险标注]
   → Registry
   → engine / parser / harness               [Phase 1 原样复用]
```

## 4. 映射规则(`ha_mapping`)

### 4.1 v1 纳管的域(其余实体过滤掉)

`light` · `switch` · `climate` · `cover` · `lock` · `alarm_control_panel` · `fan` · `valve`

传感器、`binary_sensor`、`sensor`、`device_tracker` 等只读/不可操作实体一律跳过。无法识别域或缺关键字段的实体:**跳过 + 记一条 warning,不崩**。

### 4.2 实体 → Device 字段

- `device_id` = HA `entity_id`(如 `light.living_room`)
- `name` = `attributes.friendly_name`(缺则用 entity_id)
- `type` = 域(entity_id 的前缀)
- `area` = 暂留空(area 走注册表,v1 不取)
- `operations` = 该域支持的服务(见下),参数范围从属性/服务字段映射

### 4.3 操作与参数范围(按域)

| 域 | 操作(我们的 operation) | 参数 → 范围来源 |
|---|---|---|
| light | turn_on / turn_off | `brightness_pct` 0–100(若实体 `supported_color_modes` 支持亮度) |
| switch | turn_on / turn_off | 无 |
| climate | set_temperature / set_hvac_mode / turn_on / turn_off | `temperature` ← 属性 `min_temp`/`max_temp`;`hvac_mode` ← 属性 `hvac_modes`(enum) |
| cover | open_cover / close_cover / set_cover_position | `position` 0–100 |
| lock | lock / unlock | 无 |
| alarm_control_panel | alarm_arm_home / alarm_arm_away / alarm_disarm | 无 |
| fan | turn_on / turn_off / set_percentage | `percentage` 0–100 |
| valve | open_valve / close_valve / set_valve_position | `position` 0–100 |

取不到精确范围的参数 → 该参数不带范围(validator 跳过范围检查,但仍查类型/枚举/存在性)。

### 4.4 危险默认表(操作级)+ 用户覆盖

内置默认(其余一律 `dangerous=false`):

| 域 / 条件 | 危险操作 |
|---|---|
| lock | `unlock`、`open` |
| alarm_control_panel | `alarm_disarm` |
| cover 且 `device_class ∈ {garage, gate, door}` | `open_cover`、`set_cover_position` |
| valve | `open_valve`、`close_valve`、`set_valve_position` |

**用户覆盖** `data/ha_overrides.json`(可选):按 `entity_id` → `{operation: dangerous(bool)}` 精确覆盖任意条目(既能加严也能放宽)。覆盖优先级高于默认表。

> 注:真实"燃气阀"在 HA 里多半是 `switch` 或 `valve`。若是普通 `switch`(默认安全),用户用覆盖文件把它标危险——这正是覆盖机制存在的理由,也是 D4 的兜底。

## 5. 错误处理(fail closed 延续)

| 故障 | 处理 |
|---|---|
| HA 不可达 / 超时 | `ha_client.fetch()` 抛清晰错误;**不以空/残缺清单静默启动**(没有设备就无法安全决策) |
| 认证失败(token 无效) | 抛清晰错误,提示检查 token |
| 单个实体字段畸形/域未知 | 跳过该实体 + warning,不影响其余 |
| `/api/services` 缺某服务定义 | 该操作用内置默认参数范围;仍可用 |

## 6. 测试(样本数据)

- `tests/fixtures/ha_states.json`、`ha_services.json`:手工录一份**真实格式**的样本,覆盖全部 v1 域 + 各危险场景(门锁、安防、车库门 cover、阀门、普通插座)。
- 单测断言 `map_ha(...)` 产出:
  - 正确的 `Device`(id / name / type / operations / 参数范围);
  - 危险标记正确(`lock.unlock`=危险、`lock.lock`=安全、garage `cover.open_cover`=危险、curtain cover=安全、`valve.*`=危险、普通 `switch`=安全);
  - 用户覆盖能翻转(加严 + 放宽各一例);
  - 越界/只读实体被过滤;畸形实体被跳过不崩。
- 一条集成单测:用 `Registry.from_ha(样本)` + mock parser 跑 `engine.decide`,确认上层逻辑在 HA 来源的 registry 上原样工作。
- **不联网**。真 HA 验证(连真实小米-in-HA)等环境就绪后另做。

## 7. 成功标准

- 上述映射/危险/覆盖/过滤单测全绿;
- 现有 Phase 1 全套测试仍全绿(`from_file` 与 8 设备 devices.json 不动,无回归);
- `ha_client` 是唯一碰 HA 网络的模块;`ha_mapping` 纯函数;
- 产出可被现有 `engine`/`harness` 直接消费的 `Registry`。

达成即:**我们能把任意 HA(含你的小米-in-HA)的设备清单,确定性地、带危险标注地,喂给已验证的决策内核**——为 P2.2(执行)铺好路。
