# P3.6 — 多候选消歧(设计)

- **日期**:2026-06-10
- **状态**:设计已获批,待写实现计划
- **上游**:`2026-06-04-p3-translation-layer-direction.md`;P3.2+3.3 真机 E2E 实测
- **动因(实测)**:curated 真机注册表上,「关掉卧室的灯」→ confirm(0.6)——卧室有两盏 Yeelight,模型硬选一盏低置信提交。现有 confirm 只能答"是/否",用户无法说"我要另一盏"。

## 1. 目标

歧义指令从"硬猜 + 是/否确认"变成"**列候选 → 问哪一个 → 选择后确定性执行**"。选择环节不再过 LLM(安全门不赌采样,Phase 1b 教训);选中危险操作仍触发二次确认。

## 2. 契约(`models.py`)

- `ParseResult.candidates: list[str] = []`(候选 device_id 列表)。
  语义:**`candidates` 非空 ⇒ 歧义**,此时 `device_id` 应为空,`operation`/`params`/`confidence` 照常填。
- `Decision.candidates: list[str] = []`(过滤后的有效候选,供上层渲染)。
- `Stage` 新增 `"ambiguous"`。`Verdict` 不变(歧义用 `confirm`)。

## 3. 引擎(`engine.py`)

`decide()` 在 recognized 检查之后、feasibility 之前加歧义关卡:

1. 若 `parse.candidates` 非空:先**确定性过滤**——候选必须在注册表存在且支持 `parse.operation`;
2. 过滤后 ≥2 → `Decision(verdict="confirm", stage="ambiguous", candidates=有效候选, reason="多台设备匹配,需要选择")`,**跳过 τ 关卡**(歧义本身就是问题,不再叠加置信度疑问);
3. 过滤后 =1 → 当作普通解析:以该唯一候选为 `device_id`,继续走 feasibility → τ → safety 全部关卡;
4. 过滤后 =0 → `reject(stage="parse")`(模型给的候选全是幻觉,等同没识别)。
5. 若模型同时填了 `device_id` 和 `candidates` → 以歧义分支为准(宁可多问,不可误选)。

新增 `decide_resolved(device_id, operation, params) -> Decision`:跳过解析与 τ,只走 feasibility + safety。
给"用户已明确选择"的场景复用四关里的确定性关卡;同样适用于将来任何"无 LLM 重审"的入口。

## 4. 控制器(`controller.py`)

- `Outcome.choices: list[str] | None = None`——`stage == "ambiguous"` 时携带候选 id。
- `handle()`:`confirm + ambiguous` → `Outcome(needs_confirmation=True, choices=候选, prompt=编号列表话术)`。
- 新增 `choose(decision: Decision, device_id: str) -> Outcome`:
  1. 所选不在 `decision.candidates` 内 → 不执行,返回错误 Outcome(防伪造选择);
  2. 合法 → `engine.decide_resolved(...)`:allow → 执行;confirm(危险操作)→ 再给确认话术(无状态链式,复用现有 `confirm()`);reject → 不执行。
- 话术渲染:通过 `self.engine.registry.get(id)` 取设备名(+area),输出
  `你是说哪一个?1) 卧室 Yeelight Screen Light Bar light 2) 卧室 ... Ambient Light`。

## 5. 提示词(`prompts.py`)

SYSTEM_PROMPT 增加一条规则:
"若指令含糊地匹配多个设备(同名/同区域/同类型),不要硬猜:把全部可能的 device_id 列入 candidates,device_id 留空;operation 与 params 照常填。"

## 6. 错误处理

- 候选含不存在设备/不支持操作 → 引擎确定性过滤(§3),不信任模型自由文本;
- `choose` 收到候选外的 id → 拒绝执行;
- 选择后的危险操作 → 必须再过 safety 确认,不因"用户已经选过一次"而放行。

## 7. 测试(两层)

- **纯单元**(FakeParser 注入):候选过滤(幻觉剔除);=1 降级为普通解析且继续过 τ/safety;=0 reject;≥2 confirm(ambiguous) 且不触发 τ;device_id+candidates 同时存在时歧义优先;`decide_resolved` 过 feasibility/safety;`choose` 合法执行、候选外拒绝、选中危险操作 → 链式 confirm。
- **真机 E2E**(只调 `engine.decide`,不执行,避免误动真实设备):「关掉卧室的灯」→ stage=ambiguous 且两盏 Yeelight 都在候选——同时实证 gemma4 驾驭新 schema。
