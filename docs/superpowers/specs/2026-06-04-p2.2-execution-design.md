# P2.2 — 执行层 设计文档

- **日期**:2026-06-04
- **状态**:已通过设计评审,待实现
- **所属**:Phase 2 第 2 个子项目;前序 P2.1(实时设备清单)已合并 `main`

## 1. 目标

把 `allow` 的决策**真正执行**——调 HA 的 service-call API,把"分类器"变成"控制器"。`confirm` 等用户点头(P2.3);`reject` 不动作。

## 2. 设计决策(评审锁定)

- 执行走 HA REST service-call;**HA 网络边界仍只在 `ha_client`**(读 + 写都在这一处)。
- 新增一个**薄 `controller`** 串起 `engine` + 执行。
- **仅 `allow` 执行**;`confirm`/`reject` 不执行。
- **fail-safe**:执行报错如实记录,**绝不谎报成功**。
- 对着 **stub 测试**;真 HA 联调等网络环境就绪(P2.1 已备好 `from_ha`)。

## 3. 模块

| 模块 | 加什么 | 接口 |
|---|---|---|
| `ha_client`(改) | 写操作 | `call_service(domain, service, entity_id, params=None)` → POST `/api/services/<domain>/<service>`,body `{entity_id, **params}`,`raise_for_status` |
| `controller`(新) | 编排 | `Controller(engine, ha_client).handle(instruction) → Outcome{decision, executed, error}` |

## 4. 流程

```
controller.handle(指令)
  → engine.decide()            # Decision(P1 四关,已 fail-closed)
  → verdict == "allow" ?
       是 → ha_client.call_service(domain, operation, device_id, params)
       否 → 不执行
  → Outcome{decision, executed, error}
```

映射:`domain = device_id.split(".")[0]`、`service = operation`、`data = {entity_id: device_id, **params}`(操作名即 HA service 名,P2.1 已对齐)。

## 5. 错误处理

- `call_service` 抛错(HA 拒绝/网络) → `Outcome(executed=False, error=str(e))`;**失败时绝不 `executed=True`**。
- `decide()` 内部已 fail-closed;`controller` 仅在 `allow` 时执行,异常/确认/拒绝都不动 HA。

## 6. 测试(stub,零网络)

- `allow` → `call_service` 以正确的 `domain/service/entity_id/params` 被调用、`executed=True`。
- `confirm`、`reject` → `call_service` **未被调用**、`executed=False`。
- 执行抛错 → 被捕获、`executed=False`、`error` 有值。
- 用 mock parser + stub `ha_client`(注入)。

## 7. 成功标准

- 上述 stub 测试全绿;现有全套测试零回归;`ha_client` 仍是唯一碰 HA 网络的模块(POST 也在这)。
- 产出:`controller.handle()` 能把"听懂→把关→(安全则)执行"一条龙跑通(对 stub)。

## 8. 非目标(留后)

`confirm` 的执行闭环(P2.3)· 真 HA 联调(待网络)· 多步/批量执行 · 执行后回读校验状态 · 打包(P2.4)。
