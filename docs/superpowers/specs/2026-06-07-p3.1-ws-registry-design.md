# P3.1 — WebSocket 注册表升级(设计)

- **日期**:2026-06-07
- **状态**:设计已定,待写实现计划
- **上游**:`docs/superpowers/specs/2026-06-04-p3-translation-layer-direction.md`(P3 方向 + 6 块拆解 + 地基决策)

## 1. 目标

接入真小米后,`from_ha` 把 353 实体映射成 78 个"可控设备",但绝大多数是噪声(子功能开关 + 同硬件 `_2` 重复)。干净的策展信号 `entity_category` 在 REST `/api/states` 里**全是 `None`**(已实测)。

P3.1 通过 HA **WebSocket** 一次性拉取三张注册表,把策展/去重/作用域所需的元数据**挂到每个 `Device` 上**——但**不做任何过滤、去重、改名**(那是 P3.2–P3.6)。

成功标准:
1. 真机上 `entity_category` 在部分实体上为非 `None`(证明前提成立);
2. 升级后设备集合与当前 `from_ha` **完全一致**(零回归),只是多了元数据;
3. 现有全套测试零回归。

## 2. 地基决策(沿用方向文档 + 本轮 brainstorm)

- **同步 WebSocket**。P3.1 只需一次性快照,不需要实时订阅。用同步 WS(`websocket-client` 库)避免 `async/await` 扩散进 `from_ha` / `Registry` / `Controller`——全仓库保持同步。实时订阅留到以后(届时再评估)。
- **REST + WS 混合**。保留现有 `ha_client` 的 REST(states/services);新增 WS 一次性抓注册表。
- **Fetch + enrich**。P3.1 不止拉数据,还把元数据挂到 `Device` 上,使 P3.2/P3.3 成为纯逻辑改动(无需再加管线)。

## 3. 组件与边界

| 组件 | 职责 | 位置 |
|---|---|---|
| `HAClient.fetch_registries() -> tuple[list, list, list]` | 开同步 WS、HA 鉴权握手、发三条 `config/*_registry/list`、收原始 `(entities, devices, areas)` 三个列表、关连接。**纯 I/O**。 | `ha_client.py` |
| `build_registry_snapshot(entities, devices, areas) -> RegistrySnapshot` | 纯函数,把三个原始列表解析成带查找字典的小型 frozen dataclass。 | `ha_mapping.py` |
| `map_ha(states, services, overrides=None, snapshot=None)` | 新增可选 `snapshot` 参数。传入则给每个 `Device` 注入元数据;为 `None` 则**行为与今天完全一致**(向后兼容,旧测试绿)。 | `ha_mapping.py` |
| `Device` | 新增 `entity_category: str \| None = None`、`device_id: str \| None = None`;`area`(已存在,现为 `""`)被填充。 | `models.py` |

**架构不变量**:`HAClient` 仍是"唯一碰 HA 网络的模块"——WS 抓取放进 `HAClient`,join/enrich 逻辑全部留在 `ha_mapping` 纯函数里,无需网络即可测。

`RegistrySnapshot`(frozen dataclass,位于 `ha_mapping.py`):
- `by_entity: dict[entity_id, entry]`
- `by_device: dict[device_id, entry]`
- `by_area: dict[area_id, name]`

## 4. WS 协议(`fetch_registries` 内部)

1. 连 `ws(s)://<host>/api/websocket`(由 base_url 推导 scheme/host)。
2. 收 `auth_required` → 发 `{"type":"auth","access_token":<token>}` → 收 `auth_ok`(否则 `auth_invalid`,抛错)。
3. 依次发(`id` 递增):
   - `{"id":1,"type":"config/entity_registry/list"}`
   - `{"id":2,"type":"config/device_registry/list"}`
   - `{"id":3,"type":"config/area_registry/list"}`
4. 收三条 `result`,按 `id` 取 `result` 字段;关连接,返回三个列表。

## 5. enrichment join(每个实体,在 `map_ha` 内)

- `entity_category` ← entity 注册表项的 `entity_category`(`None` / `"config"` / `"diagnostic"`)
- `device_id` ← entity 项的 `device_id`
- `area` 名 ← 先取 **entity** 项的 `area_id`,回退到其 **device** 项的 `area_id`,再到 area 注册表查名;都没有则 `""`
- entity 无注册表项 → 元数据保持 `None`/`""`,**不崩**

**不过滤、不去重、不改名。** 输出仍是 78 个设备,只是带上了元数据。

## 6. 错误处理

- 鉴权失败(`auth_invalid`)→ 抛清晰错误(对齐 REST 的 `raise_for_status()`,在 fetch 时 fail loud)。
- WS 连接失败 / 超时 → 异常向上抛,不吞。
- 单条命令 `success: false` → 带 HA 错误消息抛出(缺权限等可见,而非静默空列表)。
- 单条注册表项畸形 → 同 `map_ha` 现有姿态:跳过 + `logger.warning`,不拖垮整张快照。

## 7. 测试(三层)

- **纯单元**(`build_registry_snapshot` + `map_ha` join):手搓 fixture。覆盖 area 解析回退(entity `area_id` → device `area_id` → 无)、无注册表项实体 → 元数据 `None`、`config`/`diagnostic` 类别被挂上(而非被滤掉)。无网络。
- **Stub WS**(`HAClient.fetch_registries`):假 socket 对象重放鉴权握手 + 三条命令响应,验证协议循环(auth 消息、`id` 递增、result 收集),无需真服务器。
- **真机**(本台 MacBook 的 HA):抓一次实快照,断言 (a) 部分实体 `entity_category` 非 `None`——证明前提;(b) 设备集合与当前 `from_ha` 一致(零回归),只是被 enriched。
