# P3.2+P3.3 — 策展去噪 + 去重(设计)

- **日期**:2026-06-10
- **状态**:设计已获批,待写实现计划
- **上游**:`2026-06-04-p3-translation-layer-direction.md`(含 5.5 产品形态决策)、P3.1 设计
- **红线**:零配置默认可用。本设计不引入任何用户配置文件。

## 1. 目标

P3.1 后真机现状:78 个"可控设备"里,小米侧几乎全是噪声——**同硬件 `_2` 镜像**(设备挂两个米家"家",成对导入)+ **子功能设置开关**(提示音/ECO/推送/水印/固件升级…)。后果:"打开空调"reject。

本块用纯逻辑把目录收敛成"真实可控设备",预期 78 → 约 30,且"打开空调"命中唯一 climate 实体。

## 2. 真机钉死的两个事实(设计依据)

1. **去重钥匙是确定性的**:重复设备对的 device registry `identifiers` 形如
   `['xiaomi_miot', '<MAC>-<config_entry_id>']`,同 MAC、不同 config entry。
   把 identifier 值里本设备 `config_entries` 的 id 后缀剥掉 → 硬件唯一键。
2. **从属开关有结构信号**:门铃/摄像机/扫地机/音箱/电视的设备组里都有"主域"实体
   (camera/media_player/vacuum…,多在我们不支持的域里所以之前不可见)。
   设备有主域实体 → 它的 switch 全是子功能;设备只有 switch → switch 即设备本身(电蚊香)。

## 3. 组件与边界

新建 **`gatekeeper/curation.py`**(纯函数模块,无 I/O):

```
curate(devices: dict[str, Device], snapshot: RegistrySnapshot) -> dict[str, Device]
```

内部两步,顺序固定:**先去重,后策展**(去重在设备粒度,策展在实体粒度)。

接线:`Registry.from_ha(states, services, overrides, snapshot)` 在 `snapshot is not None` 时
自动对 `map_ha` 结果应用 `curate`——零配置默认开。需要未策展原始映射的场景(对照实验、
调试)直接调 `map_ha`。`map_ha` 本身不变。

## 4. 去重(P3.3)

- **硬件键**:对每个 device registry 条目,取其 `identifiers` 中每个 `(domain, value)`;
  若 `value` 以 `-<本设备某 config_entry id>` 结尾则剥掉该后缀 → 规范化键。
  任一规范化键相同的设备归为一组。无 `identifiers` 的设备不参与去重。
- **选代表**:组内选"实体 id 带 `_<数字>` 后缀最少"的设备;平手取 device id 字典序最小。
  确定性,不依赖名字。
- **执行**:非代表设备的全部实体从目录剔除。`device_id is None` 的实体(demo 等)不受影响。
- **附带收益**:两个米家"家"折叠成一个,P3.5(作用域)的最大动因被部分消解。

## 5. 策展(P3.2)

对去重后的目录:

1. **丢 `entity_category ∈ {config, diagnostic}`**(真机上即 indicator light)。
2. **从属开关规则**:取该实体所属物理设备在 snapshot 里的**全部**注册实体
   (含不支持的域;域取 entity_id 前缀;忽略 config/diagnostic 的);
   若存在主域实体(`PRIMARY_DOMAINS = {camera, media_player, vacuum, climate, light,
   cover, lock, fan, valve, alarm_control_panel, water_heater, humidifier}`,不含 switch)
   → 该设备的所有 `switch` 实体剔除;
   若设备的可控实体只有 switch → 保留(switch 即设备)。
   sensor/binary_sensor/number/select/button/update 不算主域(它们是读数/设置)。
- **已确认的默认取舍**:空调"上下摆风"、摄像机"摄像机控制开关"等被默认隐藏;
  个别捞回交给 P3.4 的覆盖机制,本块不做。
- 非 switch 域之间互不压制(Yeelight 主灯+氛围灯都保留)。

## 6. 真机预期(验收对照)

- 78 → 约 30;无任何幸存的 `_N` 镜像对(同硬件键只剩一份);
- 门铃 9、摄像机 6 个子开关全部消失;电蚊香保留;
- "空调插座"组只剩 **1 个** climate 实体(摆风 switch 隐藏)——"打开空调"的目标唯一;
- demo 实体(无 device_id)原样通过。

## 7. 错误处理

纯逻辑同 `ha_mapping` 姿态:identifier 结构异常(非 `[domain, value]` 对)→ 该设备
视为无 identifier(不去重),不崩;策展遇缺失字段一律按"信息不足→不动它"处理。
宁可少删,不可误删。

## 8. 测试(两层)

- **纯单元**(手搓 fixture):后缀剥离(MAC 含 `:`、值内含 `-`);无 identifier 不去重;
  代表选择(无后缀者胜、平手 id 序);镜像实体被剔;config/diagnostic 被丢;
  camera 设备的 switch 被压制;switch-only 设备保留;climate 设备压 switch 留 climate;
  非 switch 域互不压制;畸形 identifier 不崩。
- **真机**:`harness/p3_ws_snapshot.py` 升级为输出 curate 前后对比,断言第 6 节四条。
